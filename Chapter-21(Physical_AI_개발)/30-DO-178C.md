## DO-178C
DO-178C는 항공기 탑재 소프트웨어의 안전성과 신뢰성을 보장하기 위한 국제 표준입니다.  
항공기 소프트웨어가 인증을 받기 위해 따라야 하는 개발·검증 절차를 규정하며, FAA(미국 연방항공청),  
EASA(유럽항공안전청) 등 주요 항공 당국이 인정하는 기준.

## 📌 DO-178C 기본 개념
- 정식 명칭: Software Considerations in Airborne Systems and Equipment Certification
- 발행 기관: RTCA(미국)와 EUROCAE(유럽) 공동 작업
- 출시 시점: 2012년 (DO-178B의 개정판)
- 적용 범위: 항공기 비행 제어, 항법, 통신, 감시 등 모든 소프트웨어 기반 시스템

## 📊 Design Assurance Levels (DAL)
DO-178C는 소프트웨어 실패가 항공 안전에 미치는 영향에 따라 5단계 DAL을 정의합니다:
| 수준  | 실패 영향                                      | 요구 사항                                   |
|-------|-----------------------------------------------|---------------------------------------------|
| DAL A | Catastrophic (항공기 손실, 다수 사망)          | 가장 엄격한 검증, MC/DC 코드 커버리지, 형식적 방법 |
| DAL B | Hazardous / Severe-Major (심각한 부상, 안전 여유 감소) | 높은 수준의 테스트와 문서화                  |
| DAL C | Major (승객 불편, 승무원 업무 증가)            | 표준적인 검증 필요                          |
| DAL D | Minor (작은 영향, 경미한 불편)                 | 제한된 검증                                 |
| DAL E | No Effect (안전 영향 없음)                     | 인증 요구 없음                              |


## 📌 DO-178C의 주요 특징
- 요구사항 기반 개발: 모든 소프트웨어 기능은 명확한 요구사항에서 출발해야 함
- 추적성(Traceability): 요구사항 ↔ 설계 ↔ 코드 ↔ 테스트 간 연결을 반드시 문서화
- 검증 활동: 코드 리뷰, 테스트, 커버리지 분석 등 체계적 검증 필수
- 추가 문서들:
  - DO-330 (툴 인증)
  - DO-331 (모델 기반 개발)
  - DO-332 (객체지향 기술)
  - DO-333 (형식적 방법)

## 🎯 요약
- 👉 DO-178C는 항공기 소프트웨어 개발과 인증을 위한 국제 표준으로, 실패 영향에 따라 DAL A~E까지 등급을 나누고, 각 등급에 맞는 개발·검증 절차를 요구합니다.
- 항공기 비행 제어 같은 핵심 시스템은 DAL A로 분류되어 가장 엄격한 검증을 거쳐야 하며, 이는 항공 안전을 위한 필수 절차입니다.

# 검증
DO-178C에서 DAL A로 분류된 소프트웨어는 항공 안전에 치명적인 영향을 줄 수 있기 때문에, 가장 엄격한 검증 절차를 거칩니다.

📌 DAL A 검증 절차 (핵심 단계)
- 요구사항 검증 (Requirements Verification)
  - 모든 기능은 명확한 요구사항에서 출발해야 함
  - 요구사항이 완전성, 일관성, 정확성을 갖추었는지 검증
- 설계 검증 (Design Verification)
  - 고수준 설계 ↔ 저수준 설계 간 추적성 확보
  - 설계가 요구사항을 충족하는지 확인
- 코드 검증 (Source Code Verification)
  - 코드가 설계와 요구사항을 충족하는지 검증
  - 코딩 표준 준수 여부 확인 (MISRA-C 같은 규칙 적용 가능)
- 테스트 검증 (Testing Verification)
  - 요구사항 기반 테스트: 모든 요구사항이 테스트로 검증되어야 함
  - 커버리지 분석: DAL A는 **MC/DC (Modified Condition/Decision Coverage)**까지 요구
    - 모든 조건과 분기 조합이 실행되었는지 확인
- 추적성(Traceability)
  - 요구사항 ↔ 설계 ↔ 코드 ↔ 테스트 간 완전한 연결 문서화
  - 누락이나 불일치가 없어야 함
- 형식적 방법(Formal Methods, 선택적)
  - 수학적 증명 기법을 활용해 요구사항과 설계의 정확성을 보장
  - DAL A에서는 권장되며, 복잡한 시스템에서 자주 사용
- 독립적 검증(Independent Verification & Validation, IV&V)
  - 개발팀과 별도의 독립된 팀이 검증 수행
  - 객관성을 확보하기 위해 필수적

## 📊 DO-178C DAL A vs 다른 등급 비교

| 검증 단계        | DAL A                                      | DAL B                          | DAL C/D                      |
|------------------|--------------------------------------------|--------------------------------|------------------------------|
| 요구사항 검증    | 모든 요구사항 철저히 검증                   | 필수                           | 필수                         |
| 설계 검증        | 고수준 ↔ 저수준 설계 추적성 완전 확보        | 필수                           | 일부                         |
| 코드 검증        | 코드 표준 준수 + 요구사항/설계와 일치 확인   | 필수                           | 일부                         |
| 테스트 커버리지  | **MC/DC (Modified Condition/Decision Coverage)** | Decision Coverage              | Statement Coverage           |
| 독립적 검증 IV&V | **필수 (독립된 팀 검증)**                   | 권장                           | 선택                         |
| 형식적 방법      | 권장 (복잡 시스템에서 사용)                 | 선택                           | 거의 사용하지 않음           |



## 📌 요약
- 👉 DAL A 소프트웨어는 요구사항부터 코드까지 모든 단계에서 철저한 검증을 거치며, 특히 MC/DC 커버리지와 독립적 검증이 핵심입니다.
